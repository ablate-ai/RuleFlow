# 架构设计

本文档介绍 RuleFlow 的系统架构、分层设计和核心模块。

## 系统架构概览

RuleFlow 采用经典的**分层架构**（Layered Architecture），从上到下依次为：

```
┌─────────────────────────────────────────────────────────┐
│                     Web 控制台 (HTML/JS)                 │
├─────────────────────────────────────────────────────────┤
│                   HTTP API 层 (api/)                     │
├─────────────────────────────────────────────────────────┤
│                  服务层 (services/)                      │
├─────────────────────────────────────────────────────────┤
│              核心业务逻辑层 (internal/app/)              │
├─────────────────────────────────────────────────────────┤
│           数据访问层 (database/)  +  缓存层 (cache/)    │
├─────────────────────────────────────────────────────────┤
│              PostgreSQL  +  Redis                         │
└─────────────────────────────────────────────────────────┘
```

## 分层详解

### 1. 表现层（Web 控制台）

- **技术栈**: 原生 HTML + CSS + JavaScript
- **特点**: 无需构建工具，直接部署
- **功能**: 可视化管理订阅、节点、模板、配置策略

### 2. HTTP API 层（api/）

**职责**:
- 路由注册和分发
- 请求参数验证
- 响应格式化
- 中间件处理（鉴权、CORS、日志、恢复）

**关键中间件**:
```go
// main.go:126
api.LoggingMiddleware(
    api.CORSMiddleware(cfg.CORSAllowedOrigins)(
        api.RecoveryMiddleware(r)
    )
)
```

**路由结构**:
- 公开路由: `/subscribe`, `/convert`, `/health`, `/rulesets/{name}`
- API 路由: `/api/*` (需鉴权)
- Web 页面: `/dashboard`, `/subscriptions`, 等 (需鉴权)

### 3. 服务层（services/）

**职责**:
- 业务逻辑编排
- 事务协调
- 调用数据访问层
- 调度定时任务

**服务列表**:

| 服务 | 职责 |
|------|------|
| `SubscriptionService` | 订阅源 CRUD、缓存管理 |
| `NodeService` | 节点 CRUD、批量操作 |
| `TemplateService` | 模板 CRUD |
| `ConfigPolicyService` | 配置策略 CRUD、配置生成 |
| `RuleSourceService` | 规则源 CRUD |
| `SubscriptionSyncService` | 订阅同步（拉取节点） |
| `RuleSourceSyncService` | 规则源同步 |
| `MaintenanceService` | 维护操作（ID 迁移等） |

### 4. 核心业务逻辑层（internal/app/）

这是项目的核心，包含所有协议解析和配置生成逻辑。

#### 4.1 协议解析器（parser.go）

支持的协议:
- **Trojan**: `trojan://`
- **VMess**: `vmess://` (Base64 编码 JSON)
- **VLESS**: `vless://`
- **Shadowsocks**: `ss://`
- **Hysteria2**: `hysteria2://` 或 `hy2://`
- **TUIC**: `tuic://`
- **AnyTLS**: `anytls://`

解析流程:
```
订阅内容
    ↓
检测格式 (Clash YAML / Base64 / 纯文本)
    ↓
提取协议 URL
    ↓
按协议分发解析
    ↓
统一 ProxyNode 结构
```

#### 4.2 配置生成器

| 生成器 | 目标客户端 | 输出格式 |
|--------|-----------|---------|
| `config_builder.go` | Clash Meta / Stash | YAML |
| `surge_builder.go` | Surge | INI |
| `singbox_builder.go` | Sing-Box | JSON |

**Clash/Stash 配置生成流程**:
```
模板 YAML
    ↓
解析为 yaml.Node (保留格式)
    ↓
注入 proxies 节点列表
    ↓
处理 proxy-groups (filter, exclude-filter, dialer-proxy)
    ↓
注入 dialer-proxy 到对应节点
    ↓
调整目标客户端特定配置
    ↓
输出生成的 YAML
```

### 5. 数据访问层（database/）

采用 **Repository 模式**，每个实体对应一个 Repository。

**实体关系**:
```
Subscription (订阅源)
    ├─── has many → Node (节点)
    └─── used by → ConfigPolicy (配置策略)

Template (模板)
    └─── used by → ConfigPolicy (配置策略)

ConfigPolicy (配置策略)
    ├─── belongs to → Subscription
    ├─── belongs to → Template
    └─── has many → ConfigAccessLog (访问日志)
```

### 6. 缓存层（cache/）

**缓存策略**:
- 缓存键: `policy:{policyID}:config`
- TTL: 默认 3600 秒（可配置）
- 穿透保护: 数据库未命中时不缓存空值
- 主动失效: 策略更新、节点同步时清除缓存

## 核心模块设计

### 1. 配置策略生成

```go
// 调用链
GenerateConfig (api/handlers.go)
    ↓
ConfigPolicyService.GenerateConfig
    ↓
获取订阅的节点 → 获取模板内容
    ↓
app.BuildYAMLFromTemplateContent / BuildSurgeFromTemplateContent
    ↓
返回配置内容
```

### 2. 订阅同步流程

```go
SyncSubscription (api/handlers.go)
    ↓
SubscriptionSyncService.Sync
    ↓
拉取订阅内容 (app.FetchSubscription)
    ↓
解析节点 (app.ParseSubscription)
    ↓
添加国家 Emoji (app.AddCountryEmojis)
    ↓
更新数据库 (NodeRepo.ReplaceBySubscription)
    ↓
清除相关策略缓存
```

### 3. 模板扩展字段处理

Clash/Stash 模板支持三个扩展字段:

| 字段 | 作用 |
|------|------|
| `filter` | 正则匹配节点名，仅匹配的节点进入该组 |
| `exclude-filter` | 在 filter 结果中再排除 |
| `dialer-proxy` | 正则匹配中转节点，实现链式代理 |

**处理逻辑**:
1. 读取扩展字段
2. 从模板中移除这些字段
3. 根据 `filter` 和 `exclude-filter` 过滤 `__NODES__` 占位符
4. 根据 `dialer-proxy` 查找中转节点
5. 将 `dialer-proxy` 注入到对应节点的配置中

## 定时任务调度

使用 `context.Context` 实现优雅停止的定时任务:

| 调度器 | 功能 | 默认间隔 |
|--------|------|---------|
| `SubscriptionScheduler` | 自动同步订阅 | 每小时 |
| `RuleSourceScheduler` | 自动同步规则源 | 每小时 |
| `LogCleanupScheduler` | 清理访问日志 | 每天 |

**实现模式**:
```go
// 启动
scheduler.Start(ctx)

// 内部使用 time.Ticker
for {
    select {
    case <-ticker.C:
        // 执行任务
    case <-ctx.Done():
        ticker.Stop()
        return
    }
}
```

## 鉴权设计

两种鉴权方式:

1. **Web 界面**: Cookie Session
   - 登录成功后设置 `session` Cookie
   - `WebAuthMiddleware` 验证 Cookie

2. **API**: Cookie 或 Password
   - `APIAuthMiddleware` 验证 Cookie
   - 也支持通过 `ADMIN_PASSWORD` 环境变量全局保护

**无密码模式**:
- 若 `ADMIN_PASSWORD` 为空，跳过鉴权
- 生产环境强烈建议设置密码
